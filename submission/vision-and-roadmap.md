# DeepFact Validator — Vision & Future Roadmap

**「医師の習慣を、社会の真実層に。」**

---

## 1. なぜ作ったか（Why）

### 医師としての原体験

医師は診断の前に必ず「出典はどこか」を確認します。治験か、症例報告か、個人の体験談か。それだけで判断の重みが変わります。

私（株式会社Liberaiz 代表・Dr.加藤）は医学博士・産業医・元院長として、一次情報まで遡る習慣を北極星としてきました。

しかし社会に目を向けると、情報摂取はその逆を行っています。AI が答えを出す速度が上がるほど、一次情報まで遡る人間は減っていきます。チャットボット型のAIは「質問された時」しか動けません。**疑問すら持てなくなった人**には届きません。

DeepFact Validator を作ったのは、この非対称を**構造で**直すためです。

### 既存サービスの限界

| サービス | 限界 |
|---|---|
| NewsGuard | 人手評価で静的・更新遅い・個人ユーザー有料 |
| Ground News | 視点を並べるだけで「関係性」は見せない |
| AllSides | 政治的左右ラベルのみ・粒度が粗い |
| ChatGPT / Gemini | 質問されないと動かない・受動的 |

これらは全て**疑問を持つユーザー**が前提です。本当に守りたいのは「疑問を持てない人」です。

## 2. 何を期待するか（What）

### 個人レベル：誤情報による意思決定を防ぐ

- 健康判断（民間療法・誇張広告）への自衛
- 投資判断（陰謀論的市場予測）への自衛
- 政治判断（プロパガンダ）への自衛

審査員が誤って高評価しがちな「インフラの装飾」ではなく、**日常で1回でも誤情報を回避できる**実害低減を狙います。

### 社会レベル：情報の信頼性を「観測可能」にする

DevOps の Observability 概念を社会の情報層に持ち込みます。

- 個人が見た情報の信頼度が**ログに残る**
- 信頼ソース辞書が**Gitで管理され継続的に改善される**（CI/CD）
- 誤判定の警告履歴が**Postmortem 形式で記録される**

「情報の品質を測る土台」を社会の共通装備にします。

### 業界レベル：ファクトチェックの構造化

人手によるファクトチェック（Snopes / 日本ファクトチェックセンター 等）の前に、**AI Agent が構造的な異常を予兆検知**する補完層になります。

## 3. どう展開していくか（How）

### Phase 1：Hackathon 提出 → 個人ユーザー無料（2026/Q3 - Q4）

- ✅ Chrome Extension（浮遊バッジ + ワンクリック分析・HITL内蔵）
- ✅ Web UI Workbench（`/workbench/` 同オリジン・サンプル4種・HITL内蔵）
- ✅ LINE Bot（友達追加だけで使える・即時返信）
- 個人ユーザーは**無料**で誰でも使える
- 信頼ドメイン辞書（125件）と煽動パターン辞書（143語）は OSS として公開

### Phase 2：B2B / 教育・医療・報道向け（2027/H1）

- API 提供（SLA保証・RateLimit緩和）
- 月額サブスク（教育機関 / 医療機関 / 報道機関）
- カスタム辞書（業界特化）
- 監査ログ／コンプライアンス対応

医師の北極星「真偽判断は一次情報まで遡る」を、医療従事者教育に応用：
- **医療系学生・看護師の情報リテラシー教材**
- **病院内 SNS共有のスクリーニング**
- **医学論文の構造異常検知**

### Phase 3：情報インフラ化（2027/H2 〜）

- 自治体・政府機関との連携可能性（地方紙の信頼度可視化等）
- 国際展開（英語・中国語・韓国語 対応）
- ブラウザ標準API化への提唱（W3C議論）
- 報道機関向け SLA 99.99%

## 5. DevOps 文脈での社会的意義

このプロダクトは**「DevOpsプラットフォームを社会の真実層に応用する」最初の試み**です。

| DevOps | 社会装備としての翻訳 |
|---|---|
| Observability | 情報摂取の可視化（誰がいつ何を見たかを構造化） |
| CI/CD | 信頼ソース辞書の Git 駆動継続改善 |
| SRE | 「真実の SLI/SLO」設定（unknown_rate / cache_hit_rate 等） |
| Postmortem | 誤情報事案を Postmortem テンプレで記録 |

「インフラを止めない技術」を「情報を腐らせない仕組み」に転用します。

## 7. リスクと社会的責任

### 「Validator が新しい権威になる」リスクへの自衛

DeepFact 自身が「真実の審判者」になることは避けます。

- **判定の根拠を必ず提示**（一次情報URL・構造観察・対立見解）
- **HITL フィードバック**でユーザーが Validator を再評価
- **判定が誤った時の「警告履歴 Postmortem」公開**
- **オープンソース化**で監査可能性を担保

### Brand Soul「品格」

発信者個人・組織を断定的に指摘することは原則として避け、構造を可視化する設計を貫きます。「○○という構造」「○○の傾向」形式のみで提示し、政治的・宗教的に微妙な主張は「複数の立場が成り立つ構造」を contrarian_views で提示します。
